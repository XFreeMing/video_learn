## 1. 系统本体定位

```
本体 = 基于「7 步问题解决法 + 动态知识图谱」的多智能体复杂问题解决平台
投标 = 场景适配层(scenario lens)的第一个特化,不是系统边界
```

铁律(来自上游设计稿):

> 每个产物都先被整理成一张**原子化知识卡片**(全局唯一名 + 溯源 + 关系)。
> 文档 / Agent 输出 / 聊天只是「来源材料」,不是知识单元。
> 卡片之间靠**显式关系**连成**动态知识图谱**。

推论:系统所有产物本质是**同一种东西——一张多态卡片**。七步法只是给卡片标注「阶段(phase)」,逻辑树 / 假设 / 任务 / 决策 / 结论都是卡片的不同 `type`。
→ 这就是 **MagicCard(魔法卡片)**:一个组件,多态渲染,关系互联。**七步法不进 UI**,只作为卡片的 phase 标注。

## 2. 项目空间(最外层容器)

项目空间是系统的**最外层容器**。一切——卡片、智能体、任务、聊天、事件——都归属于某个 `Project`。所有视图都在「当前项目」上下文中渲染。

### 2.1 边界

```text
每个复杂目标 = 一个 Project + 一个独立项目知识空间(工作记忆)
项目卡片:spaceType = project,spaceScope = project_id
企业知识空间(enterprise)是共享的:项目通过引用(SpaceCitation)使用,不直接写入
智能体在项目内注册 / 认领;外部智能体只能拿到「授权子图」,不默认看全量
```

### 2.2 入口与操作

```text
[+ 新建项目]      Header 入口 → 对应上游 /project 命令
                  /project + 招标文件
                    → createProjectSpace → ingestProjectSource → 解析块 → 场景镜头抽卡 …
[ProjectSwitcher] Header 左侧项目切换器:切换当前 Project,所有视图随之换上下文
[+ 添加外部智能体] 「智能体」视图内的操作:
                    注册 External Agent → 能力声明 → 权限 / 可访问知识子图边界
                    → 进入项目参与者注册表(Participant Registry)
```

### 2.3 项目聊天(Project Chat)

项目级聊天是**来源材料入口**,不是知识本身(铁律:聊天不直接是知识)。

```text
用户 / 智能体在项目聊天里对话
  → AgentOrchestrator 调用目标智能体
  → 输出存为 AgentOutputRecord(过程材料,完整留痕)
  → 抽取 ExtractedClaim → KnowledgeCandidate
  → 证据绑定 / 链接已有卡片 / 人工确认
  → 仅已验证候选晋升为 KnowledgeCard 进入图谱
```

形态:**跨视图常驻的可停靠面板(右侧抽屉)+ 独立路由**。聊天产物必须经 `AgentKnowledgeIntakeService` 才进知识图谱,不能直接落库为知识。

## 3. 核心原语:MagicCard

一张卡 = 统一外壳 + 按 `cardType` 多态的卡身 + 关系把手 +(可选)对话能力。

### 3.1 统一外壳(所有卡共有,对齐 `src/db/schema/knowledge.ts`)

```text
globalName   全局唯一名         ← 人和 Agent 稳定引用的标识
cardType     类型徽章           ← 决定卡身渲染
status       draft | active | archived
phase        阶段标注           ← 七步法阶段(problem_framing/structuring/...)
tags         MECE 多级标签路径
sources      溯源(来源材料 + 页码/块 + 整理人 + 验证记录 + 适用边界)
relations    关系把手           ← 见 3.3
```

### 3.2 多态卡身(覆盖「不同阶段的全部产物」)

| 产物 | `cardType` | 卡身渲染要点 | 现有 schema 锚点 |
| --- | --- | --- | --- |
| 问题定义 | `problem_frame` | 决策者 / 成功标准 / 范围 / 约束 / 精度 字段表 | `projects.context`(待补 ProblemFrame) |
| 各种逻辑树 | `logic_tree` | 树型(要素/归纳/演绎/假设/决策)+ 节点/边可视化 | `logic_trees(nodes,edges,treeType)` |
| 各种假设 | `hypothesis` | 命题 + 状态(待验证/支持/反驳/混合)+ 置信度 + 证据链 | `hypotheses` |
| 各种任务 | `task` | 公告栏卡:能力要求 / 优先级 / 风险 / 认领 | `tasks` + `TaskBoardService` |
| 各种逻辑判断 | `decision` | if-then:条件 → 判断 → 方案 → 结果 | `knowledge_cards(cardType=decision)` |
| 各种结论 | `conclusion` | 结论表达 + 支撑子图引用 | (待补) |
| 中间文档产物 | `artifact` / `deliverable_unit` | 文档预览 + 点对点证据链 | `knowledge_cards(cardType=artifact)` |
| 证据 / 能力 / 案例 | `evidence` / `capability` / `case` | 内容 + 适用边界 + 权威评分 | `knowledge_cards` |

> 同一个 `<MagicCard>` 组件,靠 `cardType` 切换卡身。
> **新增产物类型 = 加一个卡身分支,顶层布局零改动。**

### 3.3 关系(动态知识图谱的边,对齐 `knowledge_relations`)

```text
satisfies  proves  supports  refutes  depends_on  conflicts_with
derived_from  cites  applies_to  promoted_to  replaces  version_of
```

可解释子图示例(`KnowledgeGraphService.explainRelationPath`):

```text
Requirement R
  -> requires  Capability C
  -> proven_by Evidence E1 / E2
  -> supported_by Case A
  -> constrained_by Risk X
  -> expressed_as DeliverableUnit D
```

### 3.4 对话卡片(可微调的活产物)

MagicCard 的一种**交互能力**(不是单独 cardType,而是产物类卡片 `artifact` / `deliverable_unit` / `conclusion` 可开启的模式)。

```text
对话卡片 = 卡片 + 绑定到「该卡的产出智能体」的对话线程
```

- 卡片内嵌一个对话线程,可**直接对这张产物卡聊天**,细粒度微调内容(改写一段、补一条证据、调整语气、收紧承诺)。
- 对话作用域 = **这张卡(card-scoped)**,不污染项目级聊天上下文。
- 每轮微调产生**卡片新版本**(`content` 版本化),`sources` 溯源与 `relations` 关系保持不变。
- 微调若产生新断言 / 新证据,仍走 `AgentKnowledgeIntakeService` 入图谱,不绕过审核。

**与项目聊天的区别:**

| 维度 | 项目聊天(2.3) | 对话卡片(3.4) |
| --- | --- | --- |
| 作用域 | 整个项目(来源材料入口) | 单张产物卡 |
| 目的 | 接入材料 / 补企业知识 / 广义协作 | 微调某个具体产物 |
| 产出 | AgentOutputRecord → 候选 | 卡片新版本(就地精修) |
| 上下文 | 项目知识子图 | 该卡 + 其证据链子图 |

价值:产物不再是静态结果,而是**可对话的「活产物」**——通过与智能体对话逐步收敛到可用,正好支撑上游 58「标书正文写作工作台 + 点对点证据链审查」。

## 8. 草图吸收的具体形态

> 来源:用户手绘草图两张(事件卡片流 / 个人问题区-空间)。
> 草图在四处比前文更具体,本节将其固化,并明确与已有章节的对应。

### 8.1 事件即卡片:中枢事件带(升级 §4「事件流」)

草图的橙色「事件卡片」横带揭示:**事件不是一个并列 tab,而是贯穿全局的中枢脊柱**。
上方各 Agent 把领域事件**发射到带上**,下方用户交互**挂接到带上**。这正是 Outbox → EventBus 的 UI 化身。

```
   问题拆解Agent  假设Agent  数据任务Agent  任务处理Agent  假设验证Agent  阶段分析Agent  复合方案
        │            │           │             │              │            │           │
        ▼            ▼           ▼             ▼              ▼            ▼           ▼
 ┌─────────────────────────────  事件卡片带(中枢)  ──────────────────────────────────┐
 │ 问题已更新 · 假设已更新 · 数据任务已创建 · 数据任务已处理 · 假设已验证 · 已分析 · 方案已生成 │
 └──────────────────────────────────────┬──────────────────────────────────────────┘
              ▲(卡片交互事件)            │(用户行为)            ▲(Chat)
        可交互卡片模块 ───────────────  用户行为  ─────────────── 聊天模块
```

要点:

```text
- 事件命名 = 过去式领域事件,直接作为 outbox event.type 枚举候选:
    ProblemUpdated / HypothesisUpdated / DataTaskCreated / DataTaskProcessed
    HypothesisValidated / Analyzed / SolutionGenerated
- 事件带是「视图脊柱」:总览页常驻顶/侧,其它视图可折叠引用同一事件源。
- 上游(Agent 发射)与下游(用户交互回挂)共享同一条事件带 = 单一事实源。
```

### 8.2 两条 WebSocket:按系统边界划分(细化 §5 实时传输层)

草图画了多路 WS;落地按**系统边界**(谁在另一端)划分为**两条**连接,而不是按内容类型拆。
内容类型(列表/详情/聊天)在 WS-1 上用**逻辑频道(channel)**多路复用即可,无需各自建连。

```
┌ 浏览器 ┐                                   ┌ 本系统服务端 ┐                 ┌ OpenClaw ┐
│        │ ── WS-1 内部事件总线(/ws)──────▶ │ Outbox→Bus  │ ── WS-2 Runtime ▶ │  Runtime │
│        │ ◀───── channel: card-list /     │ 中转/留痕    │ ◀──── token 流 ── │ Connector│
│        │        card-detail / chat ──────│ AgentOutput │                  └──────────┘
└────────┘                                   └─────────────┘
```

#### WS-1 内部事件总线(浏览器 ↔ 本系统)

承载本系统领域事件,源头是 Outbox → EventBus。内容类型用 channel 分流:

```text
client → server
  { op: "subscribe",   channels: ["card-list", "card-detail:<cardId>", "chat:<threadId>"] }
  { op: "unsubscribe", channels: [...] }

server → client
  { channel, type, payload, seq }
    channel = card-list | card-detail | chat   ← 逻辑频道(草图三路的逻辑等价)
    type    = 领域事件名(CardCreated / CardUpdated / ChatToken ...)
    seq     = 单调递增序号(断线重连按 seq 增量补投)
```

逻辑频道职责:

```text
card-list     列表级增量(新增 / 状态变更 / 计数)→ 总览 / 卡片空间 / 黑板
card-detail   单卡级增量(内容版本 / 关系变化)→ 当前打开的 MagicCard(按 cardId 订阅)
chat          项目聊天 + 对话卡片会话流(转发自 OpenClaw 的 token 流)
```

为什么内容类型用单连接多路复用(而非各建一条):

```text
- 连接成本:1 次握手 / 1 份心跳 / 1 次鉴权,不占浏览器并发配额。
- 顺序保证:同连接天然有序 —— 对「事件中枢带」(§8.1)不丢/有序尤为关键。
- 重连退避:1 套状态机;后端 1 个订阅集合按 channel 投递。
```

#### WS-2 OpenClaw Runtime(本系统 ↔ OpenClaw)

独立连接,因为另一端是**外部 Runtime Connector**(不同系统边界):

```text
- 承载:智能体运行时 token 流、RuntimeRun 状态、工具调用进度。
- 绑定:RuntimeSession(对话卡片 §3.4 / 项目聊天 §2.3 各自的会话)。
- 中转留痕:OpenClaw 的流不直接灌给浏览器,而是经本系统服务端中转,
            存为 AgentOutputRecord(过程材料),再由 WS-1 的 chat 频道下发前端。
- 边界意义:外部 Runtime 可替换 / 多实例 / 独立重连,不影响 WS-1 内部事件流。
```

要点:

```text
- 两条连接 = 两个系统边界:WS-1 是「本系统 ↔ 前端」,WS-2 是「本系统 ↔ OpenClaw」。
- 前端只直连 WS-1;WS-2 由服务端持有,前端永远拿到的是已留痕、已治理的事件。
- 对话卡片微调:用户输入 → WS-1 chat 上行 → 服务端经 WS-2 调 OpenClaw →
                token 流回服务端留痕 → WS-1 chat 下发;完成后经 card-detail 推新版本。
```

### 8.3 决策金字塔反馈区(「总览/结论」视图主形态)

草图二的「决策信息金字塔反馈区」= 七步法第 7 步「阐明观点」的**产物视图**(非流程步进器,与「七步法不进 UI」不冲突)。
采用 Minto 金字塔:**最终观点在顶 → 支撑结构在中 → 数据溯源做地基**,全部由 MagicCard 多态拼成。

```
┌ 个人问题区-空间(= 当前 Project 一屏)─────────────────────────────────┐
│ ┌ 唯一目标区 ┐      ┌ 用户自然语言输入区 ┐      ┌ 用户介入任务区 ┐         │
│ └───────────┘      └──────────────────┘      └───────────────┘         │
│ ┌ 决策信息金字塔反馈区 ───────────────────────────────────────────────┐ │
│ │                    ┌────── 最终观点(conclusion)──────┐               │ │
│ │            ┌ 决策树 ┐ ┌ 2-3维排序图 ┐ ┌ 方案组合 ┐ ┌ 详细计划 ┐         │ │
│ │            │decision│ │LogicNodeScore│ │synthesis │ │  plan   │         │ │
```
